# TestLens v0.2 Plan

**Status:** Draft · **Target:** `0.2.0` · **Theme:** *Ship the promised affected-tests features on a hardened, versioned core.*

This plan is derived from the v0.1.0 codebase, the explicit promises in `README.md`
(`--impact`, `--rerun`), the `docs/dogfooding.md` §7 wishlist, and the architecture
review of PR #4 (`fix/json-reporter-impact-wiring`).

> **Why 0.2.0 and not 0.1.1:** the JSON `failures[].impact` shape changed from a string
> to an object in PR #4. Pre-1.0 semver lets us bump the minor on a breaking change, and
> the JSONReport moduledoc already promises field stability across minor bumps. v0.2 is
> the moment to add `schema_version` and make that promise machine-readable.

---

## Sequencing at a glance

| Phase | Theme | Why this order |
|-------|-------|----------------|
| 0 | Harden the core | Prerequisite: makes the codebase hermetic/testable before feature work |
| V | Visual output polish | First user-facing v0.2 upgrade: make TTY/HTML output clearer and more useful |
| 1 | `--rerun` | Small, isolated quick win |
| 2 | `--impact` / affected tests | The headline feature; largest effort |
| 3 | Richness & quality gates | Polish: data, coverage, adapters |

## Phase 0 — Harden the core (do first)

### P0.1 — Thread project config explicitly; delete the global app-env side-channel

**Origin:** `@oracle` review "Blocker B1" on PR #4.

Today the Mix task, the formatter, and `Impact.find_area/2` / `Impact.classify/1`
communicate via hidden `Application` env keys (`:project_config`,
`:project_config_source_dir`). This is non-hermetic and is the reason `ImpactTest`
had to be forced `async: false` (the global env writes raced sibling async tests).

**Plan:**
- Add `source_dir: String.t() | nil` to `%TestLens.ProjectConfig{}`.
- Set `source_dir` on the struct inside `ProjectConfig.load_or_walk/1`
  (instead of `publish_source_dir/1` → app env).
- Activate the already-existing escape hatch `Impact.classify/2` by threading the
  config explicitly through the reporters: `TerminalReporter.render/4` →
  `render_failure_block/2` → `Impact.classify/2`, and `failure_to_json/2`.
- Delete: `Impact.area_config_source_dir/0`, `Impact.resolve_config/0`,
  `Mix.Tasks.Test.Lens` `:project_config` env write, `ProjectConfig.publish_source_dir/1`.
- Revert `test/test_lens/impact_test.exs` to `async: true` once hermetic.

**Net effect:** ~30 lines *deleted*, subsystem becomes hermetic, tests go async again.

### P0.2 — Version the JSON schema

**Origin:** `@oracle` review "S2".
- Add a top-level `"schema_version": "0.2"` field to **both** JSON outputs
  (`TerminalReporter.render_json/4` and `JSONReport.build/3`). Consumers branch on
  shape rather than guessing.
- Add a CHANGELOG `[Unreleased]` → `### Changed (breaking)` entry documenting the
  `failures[].impact` string→object migration.
- Bump `mix.exs` `0.1.0` → `0.2.0` (and `TestLens.version/0`'s `@version`).

### P0.3 — Dedupe the JSON stringifier

**Origin:** `@oracle` review "S1".
- `TerminalReporter.stringify_keys/1`, `stringify_value/1`, `to_string_key/1` are
  byte-identical to the copies in `JSONReport`. `JSONReport` already reuses
  `TerminalReporter.encode_json_value/1`, so the "different lifecycles" justification
  is unsound. Promote `stringify_keys/1` to a public helper (or extract a tiny
  `TestLens.JSONEncoder`) and delete the duplicate.

### P0.4 — Clean up `load_or_walk/1` and the cwd fallback

**Origin:** `@oracle` review "S3" + "S4".
- Remove the global-env side effect from `ProjectConfig.load_or_walk/1` (including the
  surprising `Application.delete_env/2` on the *miss* path). After P0.1 the source dir
  lives on the struct, so the side effect is unnecessary.
- Delete dead code: `ProjectConfig.load_walk/1` (defined, never called).
- Drop the `Path.relative_to_cwd/1` fallback in `Impact.relativize_for_areas/1` — it
  silently re-enables the exact umbrella bug the PR fixes whenever the global is unset.

## Phase V — Visual output polish

The terminal output and HTML report are part of TestLens's product surface. v0.2 should
make them feel intentional: rich, colourful, and graphical where that improves scanning,
while preserving `--no-color` correctness and self-contained HTML.

**TTY design direction:**
- Add a framed TestLens title block.
- Move the run verdict into a prominent outcome banner with pass/fail/skip counts, run
  time, seed, and a compact progress meter.
- Use semantic colour: pass green, failure red, skip amber, critical red, other amber,
  impact high/critical red, medium yellow, low blue, none dim.
- Use consistent icons (`✓`, `✗`, `⚑`) and box-drawing section rules.
- Heat-colour slow tests so the slowest items stand out without reading every number.

**HTML design direction:**
- Mirror the TTY hierarchy with a bold summary hero, cards for critical failures, and
  matching semantic colours/icons.
- Keep the report self-contained (inline CSS, no JS, no external fonts/assets) and
  print-friendly.

**Acceptance criteria:**
- `--no-color` emits no ANSI escapes and remains readable.
- `mix test.lens` output is still copy/paste friendly; commands are visually distinct.
- Existing JSON output/schema remains untouched.
- `mix format --check-formatted`, `mix compile --warnings-as-errors`,
  `mix credo --strict`, and `mix test` pass.

## Phase 1 — `--rerun` (quick win)

`--rerun` is parsed (`mix/tasks/test.lens.ex`) and `Config.rerun` exists, but neither
is wired to anything. The `TestLens.Rerun` module already computes the right command.

**Plan:**
- Wire `--rerun` to the `TestLens.Rerun` module.
- Replace `TerminalReporter.rerun_command/1`'s hardcoded
  `"mix test.lens -- --failed"` string with a call to `TestLens.Rerun`.
- Drive `Config.rerun` through the formatter so the flag actually changes behavior.
- Tests: multiple-failure case, empty-list edge (currently only 2 trivial tests exist).

**Effort:** Small.

## Phase 2 — `--impact` / affected tests (headline)

The README's marquee v0.2 promise. Two documented stubs block it:

| Stub | Current | Real behavior |
|------|---------|---------------|
| `Impact.changed_files_since/1` (`impact.ex:211`) | `[]` | Walk `git diff` since a `DateTime` / ref |
| `Impact.affected_tests/2` (`impact.ex:222`) | `[]` | Map changed source files → likely-affected test modules |

**Plan:**
- `changed_files_since/1`: shell out to `git diff --name-only <since>` (and handle the
  "no git / dirty tree / shallow clone" fallbacks safely, consistent with the
  never-raise posture of `ProjectConfig`).
- `affected_tests/2`: use the Classifier's module→file mapping (collected by the
  formatter via `module_started` events in `EventStore`) to map changed source files to
  their test modules. Document the matching heuristic (co-located `lib/` ↔ `test/`,
  suffix conventions).
- Wire the `--impact` flag to filter the test set passed to `mix test`.
- Property tests for the git-output parsing (mock the porcelain; do not depend on a
  live repo in CI).

**Effort:** Large. **Dependencies:** P0.1 (so the classifier config is hermetic).

## Phase 3 — Richness & quality gates

### P3.1 — Data richness (top dogfooding asks)
- `Result.line` is always `nil` (`ExUnit.TestModule` has no line field). Find a source
  (e.g. derive from the first failure's stacktrace / `ExUnit.Test.tags`).
- Add captured `ExUnit` log output to the JSON/HTML failure entries.

### P3.2 — Test coverage for the classifier adapters
There are **zero** direct tests for the 4 classifier adapters (`Adapters.Phoenix/Ecto/
LiveView/OTP`) and the 13 failure adapters. Add characterization tests *before* touching
any `match?/1` logic so regressions are guarded.

### P3.3 — Refresh the OTP adapter
`Adapters.OTP` matches deprecated `GenEvent`/`GenStage` and omits modern OTP 28
behaviours. Add `Agent`, `Registry`, `ETS`; keep the deprecated suffixes or drop them
per a decision.

### P3.4 — Tighten Credo
`.credo.exs` has a comment reserving the (currently disabled) style checks for v0.2.
Enable them with a stricter config.

### P3.5 — End-to-end integration test
No test currently runs `mix test.lens --json` / `--html` as a subprocess. Add one to
guard the full task → formatter → artifact path.

## Explicitly out of scope for v0.2

- Consolidating the TTY-mode JSON encoder with `JSONReport` beyond the P0.3 dedupe.
- New failure adapters beyond refreshing OTP (collect via `docs/dogfooding.md` §7 first).
- A compat matrix in CI (1.18/1.19 × OTP 27/28) — attempted and abandoned as flaky
  under `setup-beam@v1`; revisit only if a consumer needs it.
